# 第 4 课：代币交换

> 建议课时：60～90 分钟｜对应页面：`/`

## 学习目标

- 理解报价、价格影响、滑点、最小收到量和截止时间；
- 完成输入代币、查询报价、授权和交换的完整交互；
- 避免过期报价、重复提交与 decimals 错误；
- 为关键交易状态提供清晰反馈。

## 交换流程

```text
选择代币与输入数量
        ↓（防抖）
POST /api/quote → 展示报价、价格影响、最小收到量
        ↓
检查余额与 allowance
        ↓不足
approve → 等待确认
        ↓
swap → 钱包签名 → 等待回执 → 刷新余额
```

## 关键参数

| 参数 | 含义 |
| --- | --- |
| `amountIn` | 用户支付的精确输入量 |
| `quotedAmountOut` | 当前状态下的预估输出量 |
| `slippage` | 用户允许的报价偏差 |
| `amountOutMinimum` | 交易可接受的最小输出量 |
| `deadline` | 交易过期时间 |

示例：报价输出 100，滑点 0.5%，忽略舍入时 `amountOutMinimum = 99.5`。这个限制应作为链上参数，而不只是 UI 提示。

## 实操任务

1. 选择已有流动性的代币对，输入交换数量。
2. 在 Network 面板观察报价请求，解释每个返回字段。
3. 第一次交换先完成授权，确认授权对象和额度。
4. 发起交换，等待交易确认后核对两种代币余额。
5. 点击反转按钮，验证代币和输入输出状态是否正确更新。

## 前端实现要点

- 输入变化应防抖，并取消或忽略旧报价，防止响应乱序覆盖新结果。
- 禁止相同代币互换、零数量、超余额和无流动性提交。
- 报价需带生成时间；长时间未提交时重新报价。
- UI 格式化与链上整数转换必须分离，运算避免 JavaScript 浮点数。
- `approve` 与 `swap` 都应分别等待成功回执。
- 提交中锁定按钮，但交易失败后必须允许重试。

## 常见问题

- **execution reverted**：检查余额、授权、deadline、滑点和池流动性。
- **报价正常但交换失败**：链上价格可能在签名期间变化，应重新报价或合理设置滑点。
- **输出数量异常**：通常是 token 顺序或 decimals 处理错误。
- **授权后仍提示授权**：确认 spender、网络，并在回执成功后重新读取 allowance。

## 课堂检查点

- 滑点容忍度越高为什么不一定越好？
- 为什么报价结果不能保证最终成交数量？
- 如何防止较慢的旧请求覆盖最新报价？
