# 第 5 课：池子详情与历史数据

> 建议课时：45～60 分钟｜对应页面：`/pools/[address]`

## 学习目标

- 实现基于池地址的动态详情页；
- 组合链上实时状态与数据库历史事件；
- 理解价格方向、交易记录分页和统计口径；
- 完成地址校验、错误降级和定时刷新。

## 页面信息架构

```text
池子详情
├─ 基本信息：代币对、地址、费率
├─ 实时状态：当前价格、流动性、TVL
├─ 历史统计：交易量、交易次数、价格变化
├─ 最近交换：账户、方向、数量、时间、交易哈希
└─ 操作入口：交换、添加流动性
```

## 数据来源与取舍

| 数据 | 推荐来源 | 原因 |
| --- | --- | --- |
| 当前价格、流动性 | RPC / 合约读取 | 需要最新链上状态 |
| 代币符号、精度 | 合约读取并缓存 | 变化少但读取频繁 |
| 历史 Swap 事件 | 扫描器 + 数据库 | RPC 范围日志查询成本高 |
| 24h 聚合指标 | 数据库聚合 | 便于按时间窗口计算 |

索引器数据可能比链上晚几个区块，因此 UI 应标注更新时间，并允许局部数据暂时缺失。

## 实操任务

1. 从池列表进入详情页，验证 URL 中的地址。
2. 对比详情接口与链上浏览器中的池信息。
3. 查看最近交换列表，识别交易方向和 token0/token1 数量。
4. 为交易哈希、池地址和用户地址添加浏览器链接。
5. 输入非法地址和不存在的池地址，完善 400、404 与 500 状态。

## 价格方向

同一池可以展示 `Token1 per Token0` 或其倒数。页面必须明确基准代币；切换方向时，价格、涨跌幅和图表标签要同步变化。对零值求倒数前必须保护。

## 扫描器可靠性

- 持久化最后成功扫描区块，重启后从游标继续。
- 每批区块范围不宜过大，并为 RPC 限流增加重试。
- 使用 `chainId + transactionHash + logIndex` 做事件唯一键，保证重复扫描幂等。
- 考虑区块重组，生产环境可延迟若干确认后再入库。
- 聚合指标应明确时区、窗口和是否包含未最终确认事件。

## 课堂检查点

- 为什么详情页不能只依赖数据库？
- 如何唯一标识一条链上日志？
- token0/token1 对调后，价格和交易方向如何变化？

## 课后作业

为最近交换实现分页或“加载更多”，展示数据更新时间；增加价格方向切换，并为索引延迟提供友好提示。

